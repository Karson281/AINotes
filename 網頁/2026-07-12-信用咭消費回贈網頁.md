## 新增「本地餐飲」和「本地零售」消費模式 ✅

消費模式下拉選單現在有 8 個選項：

| 模式                               | 匹配欄位             | 用途           |
| -------------------------------- | ---------------- | ------------ |
| **本地餐飲**                         | `local_dining` 只 | 專門查餐廳、外賣、交通等 |
| **本地零售**                         | `local_retail` 只 | 專門查超市、藥房、零售等 |
| **本地簽帳**                         | 兩者合併             | 不分類型的本地消費    |
| 外地簽帳 / 內地簽帳 / 台灣簽帳 / 本地網購 / 外幣網購 | 不變               | —            |
選「本地餐飲」時，Travel+ VS Visa（需 $6,000 下限）在 $200 消費時不會出現，完全避免混淆。

因為 Travel+ 的本地餐飲 5% 是透過 **scenarios** 控制的，而 scenarios 有 `spend_min: 6000`。

您查詢的金額（例如 $200）未達 $6,000，所以 scenarios 的本地餐飲不觸發。

同時 `cashback.local_dining: 5.0` 雖然有值，但查詢邏輯是先用 scenarios 覆蓋，如果 scenarios 的 spend_min 不達標，就退回 `cashback` 的基礎值。

**問題根源**：`cashback.local_dining: 5.0` 讓基礎回贈率 = 5%，但 scenarios 的 spend_min 篩選只在 scenarios 層面生效，**不會阻止 cashback 基礎值被使用**。

所以 $200 查詢時：

- scenarios 的 local_dining（$6,000 下限）→ 不觸發 ✅
- cashback.local_dining = 5.0 → **直接觸發** ❌（這就是問題所在）

**解決方法（改 Gist，零 Token）：**

把 Travel+ VS Visa 的 `cashback` 裡：

JSON

```
"local_dining": 5.0,
```

改為：

JSON

```
"local_dining": 0,
```

這樣本地餐飲只由 scenarios 控制（有 $6,000 下限保護），$200 時不顯示，$6,000 時才顯示。
**總結今次的修正原則**，以後遇到類似情況可以自己處理：

如果一張卡的某個回贈**有消費下限限制**，`cashback` 欄位應設為 `0`，只在 `scenarios` 裡設定 `rate` + `spend_min`。

這樣查詢邏輯才能正確篩選，避免低消費時誤觸發。
