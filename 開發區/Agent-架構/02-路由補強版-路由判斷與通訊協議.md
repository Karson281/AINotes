# 路由補強版：雙 Agent 路由判斷標準與通訊協議

**日期：2026-07-06** | **基於實施藍圖第 1-2 章提煉**

---

## 1. Hermes 路由判斷標準（Routing Policy）

此為給 Hermes Agent 的 System Prompt，定義何時自行處理、何時轉交 Proma。

```
## 路由規則（Routing Policy）

你的型號為 DeepSeek Flash（輕量推理），適合即時查詢與數據聚合。
Proma Agent 搭載 DeepSeek Pro（深度推理），適合複雜分析。
請根據以下規則決定如何處理使用者請求：

### A. 直接處理（你回覆，不需轉交）
- 簡單事實查詢：「AE 白金卡年費幾多？」
- 已知資訊回顧：「我上次旅行去咗邊？」
- 格價查詢：「PS5 Pro 邊度最平？」
- 系統操作：「幫我 set 提醒聽日 check 機票」
- 閒聊或一般對話

### B. 查 Obsidian 知識庫後回覆（仍由你回覆）
- 需要讀取已結構化資料但無需深度推理
  - 「邊張卡海外簽賬回贈最高？」→ 查信用咭回贈.md
  - 「我上次去日本住邊間酒店？」→ 查旅行記錄/
- 注意：查知識庫後仍由你回覆，不轉交 Proma

### C. 轉交 Proma（透過 HTTP POST /task）
- 使用者的請求包含以下觸發關鍵詞：
  「分析」、「比較」、「推薦」、「策略」、「組合」、「優化」、「計劃」
- 需要個人化計算：
  - 「根據我過去三個月的消費，邊張卡最抵？」
  - 「我而家用緊邊幾張卡？有冇更好的組合？」
- 跨平台／跨類別綜合判斷：
  - 「我想去日本七日，幫我規劃大約使費同推薦信用卡組合」
- 使用者直接要求：「叫 Proma 幫我睇睇」

### D. Edge Cases
- 不確定的請求：預設先由你處理，處理不到再轉交
- 連續對話：若上一輪已轉交 Proma，同一話題延續也轉交
- 使用者明確指定：「你答就得，唔駛叫 Proma」→ 遵從
```

## 2. Hermes ↔ Proma 通訊協議

### 2.1 主要路線：HTTP over Tailscale（即時）

```
Proma HTTP Server: http://{TAILSCALE_IP}:8767
認證方式：Bearer Token (PROMA_AUTH_TOKEN)
```

#### Hermes → Proma：POST /task

```json
{
  "task_id": "credit-20260706-001",
  "type": "credit_card_analysis",
  "query": "分析我過去三個月消費，推薦信用卡組合",
  "context": {
    "user": "Karson",
    "current_cards": ["AE Platinum", "HSBC EveryMile"],
    "recent_spending": "餐飲 40%、海外簽賬 30%、網購 20%"
  }
}
```

#### Proma → Hermes：POST /result（callback）

```json
{
  "task_id": "credit-20260706-001",
  "status": "completed",
  "result": {
    "summary": "建議日常用 HSBC EveryMile，海外用 AE Platinum",
    "details": "...",
    "recommendations": [...]
  }
}
```

### 2.2 備用路線：Obsidian 檔案佇列（非同步）

當 HTTP 請求失敗（timeout / connection refused）時自動降級：

```
Hermes 寫入 → Agent-Inbox/task-001.md → git push
↓
本地 git pull → Proma 偵測新任務 → 處理
↓
Proma 寫入 → Agent-Outputs/task-001-result.md → git push
↓
VPS git pull → Hermes 偵測結果 → 推播 Telegram
```

### 2.3 雙軌切換邏輯

```
Hermes 決定轉交 Proma
  ├── HTTP POST http://{TAILSCALE_IP}:8767/task
  │   ├── 成功（202 Accepted）
  │   │   └── 等待 callback → 收到後推 Telegram
  │   │
  │   └── 失敗（connection refused / timeout）
  │       ├── 寫入 Agent-Inbox/task-xxx.md（含 callback URL）
  │       ├── 回覆使用者：「已排入分析佇列，完成後通知你」
  │       └── Proma 上線後處理 → git sync → 通知
  │
  └── 若 Proma 長期離線（多個 task 未完成）
      └── 回覆使用者：「深度分析暫時不可用，可先查基本資料」
```

## 3. Proma HTTP Server 實作規格

### 端點一覽

| Endpoint | Method | 方向 | 說明 |
|:---|:---|:---|:---|
| `/task` | POST | Hermes → Proma | 接收分析任務 |
| `/health` | GET | Hermes → Proma | 健康檢查（用於判斷是否降級） |

### 啟動配置

```bash
# Proma Windows 端
TAILSCALE_IP=100.x.x.x    # Windows 的 Tailscale IP
PORT=8767
PROMA_AUTH_TOKEN=your-token-here
HERMES_CALLBACK=http://100.x.x.x:port/result

# 只 bind Tailscale interface，不對公網暴露
python server.py --host $TAILSCALE_IP --port $PORT
```

### 健康檢查

Hermes 在轉交前可先檢查 Proma 是否在線：

```bash
curl -s http://100.x.x.x:8767/health
# 回傳: {"status": "ok"}
# 若 timeout 或非 200：降級到檔案佇列模式
```

## 4. 可選：n8n Webhook 橋接（過渡替代）

若 MQTT 持續不穩且想避免直接暴露 Proma：

```
Hermes → n8n Webhook → HTTP 轉發 → Proma (Tailscale)
         │
         └── 同時寫入 Obsidian 做 backup
```

n8n 作為可靠的中介層，但多一跳 latency。
